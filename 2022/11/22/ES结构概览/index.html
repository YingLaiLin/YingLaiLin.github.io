ES 结构概览

一个 ES Index 在集群模式下，由多个 Node （节点）组成。每个节点就是 ES 的Instance (实例)。每个节点上会有多个 shard （分片）.

与 Mongo 中 shard 的区分
这里要与 mongo 的 shard 区分, mongo 中 shard 就是节点的一种, 而在 es 中 shard 是按照集合维度定义的.

mongo: shard 上有多个集合, 每个集合又分布在不同的 shard 上.(节点维度)
ES: 某个 shard 上只有一个集合, 集合有多个分片, 对于特定的集合, 其 shard 分布在不同的 node 上.(集合维度)

ES 节点类型[1]

ES 集群中的节点类型包含:

  • master # 主节点
  • data # 通用数据节点
  • data_content # 数据目录节点
  • data_hot # 热点数据节点
  • data_warm # 中温数据节点
  • data_cold # 冷数据节点
  • data_frozen # 封存数据节点
  • ingest # 数据摄入节点
  • ml # 机器学习节点
  • remote_cluster_client # 远程候选节点
  • transform # 转换节点

注意点:

master - 主节点[2,4]

概念:主节点一般指的是集群中活跃的主节点，每个集群中只能有一个。

  • 主节点负责集群范围内的轻量级操作

    • 例如创建或删除索引、跟踪哪些节点是集群的一部分
    • 决定将哪些分片分配给哪些节点
    • 主节点必须有一个path.data 目录, 用于存储集群元数据

  • 候选主节点(master-eligible node) 可以参与主节点的选举, 其作用主要在于当主节点宕机或发生故障脱离集群时，参与选举成为新的主节点，从而避免集群无主。

  • 任何不是仅投票节点的主合格节点都可以通过主选举过程选举成为主节点

Dedicated master-eligible node 专用主节点

为了避免选出的主节点因为其他任务超载, 而将所有符合 master 条件的节点配置为专用的 master 节点, 这些节点只具有 master 角色.

符合主节点条件的节点仍然能充当协调节点, 当集群包含多个节点时,通常建议使用专用的主节点.

Voting-only master-eligible node 仅投票主节点

概念 : 仅投票主节点仍然会承担与其他主节点相同的工作, 只是其不会担任集群主节点. 和真正的主节点相比较，他们可能需要需要较少的堆和一个不太强大的 CPU. 仅投票节点的意义是在出现平票的时候投出关键票（决胜票）

由于候选主节点通常都是专用的, 由于其可能成为主节点, 因此候选主节点也不应承担 data 角色的功能, 这样会造成资源浪费. 而将其配置为仅投票节点, 让其承担 data 角色, 避免资源浪费.

主要功能：

  • 负责数据的存储和相关具体操作
    • 索引数据的创建、修改、删除、搜索、聚合
    • data节点消耗内存和磁盘IO的性能比较大。
    • 分片信息
      • 当前节点保存的分片信息
      • 每个索引在当前节点的分片信息
    • 集群元信息(用于通信)

在多层（multi-tier）部署体系结构中，数据节点可以有不同的专用角色 ：data_content，data_hot，data_warm 或 data_cold。 一个节点可以属于多个层，但是具有专用数据节点不能具有通用数据角色。

热点数据节点（Hot data node)

  • 热数据节点是时序数据的重要入口，用于存储最近频繁搜索和修改的时序数据.
  • 热层必须能够快速进行读写操作，并且需要更多的硬件资源（例如 SSD 驱动器）

中温数据节点（Warm data node）

  • 当时序数据访问频率下降，可以将其移动到中温节点.
  • 中温节点用于存储 修改较少，但仍然有查询的数据。查询的频率肯定比热点节点要少, 这些数据通常为近几周的数据.
  • 中温节点可以使用性能较低的硬件。

冷数据节点（Cold data node）

  • 当时序数据不再更新，那么可以将该数据移动到冷数据节点.
  • 冷数据节点用于存储只读，且访问频率较低的数据
  • 冷数据节点的配置可以比中温层的配置更低

封存数据节点（Frozen data node)

  • 当数据查询很少或者不再查询时, 可以将数据从 cold 移动到Fronzen节点
  • 封存层需要一个快照仓库, 以便于使用部分挂载(partially mounted indices) 的方法将数据拉取到本地. 这个本地存储在封存各节点是共用的.
  • 部分挂载其本质上就是按需去远程仓库上拉取数据并存储下来.

协调节点(Coordinating only node, 客户端节点)

主要功能：

  • 负责任务分发和结果汇聚，分担数据节点压力。

  • 无法通过配置项指定协调节点

    • 某个节点收到请求, 该节点就成为协调节点
    • 集群中的任何节点都可以是协调节点(node.roles:[])
      配置要点：

  • 大内存，最好是独占的机器

  • 协调节点对于大集群，将有益于降低主节点和数据节点负担(汇聚工作统一由专用节点处理)。

  • 但是添加过多的协调节点，将增加整个集群负担（因为需要将集群状态同步到每个协调节点）。

如上图所示, P1 P2 是主分片, R1 R2 是副本分片每个分片上对应着就是一个 Lucene Index（底层索引文件）. 也就是说, 一个ElasticSearch的Shard 本质上是一个Lucene Index。
